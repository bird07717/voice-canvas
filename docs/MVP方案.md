# AI 语音绘图工具 — MVP 方案

> 比赛题目二：纯语音控制的绘图工具。用户不能使用鼠标或键盘进行绘图创作，仅通过语音指令完成。
> 本文档记录截至目前已确定的方案，作为开发前的决策定稿。后续大脑设计（场景图 / 指令 schema / 拆解 prompt）将另行补充。

---

## 一、项目定位

不要把它理解为"一个用语音画画的小网页"，它的准确定位是：

> **一个把自然语言实时编译为结构化图形操作的、有上下文记忆和容错能力的语音交互引擎。**
> 外部框架只是它的 I/O 端口（听、画），真正的智能调度与语义理解层是自研的。

框架负责"手和眼"，自研层负责"大脑"。评分看的是大脑。

---

## 二、选题与形态

| 项目 | 确定内容 |
|---|---|
| 选题 | 题目二：AI 语音绘图工具 |
| 形态 | 纯前端 Web 应用 + 一个薄后端代理（藏 key、做模型路由） |
| 运行环境 | 本地 Windows 浏览器（Chrome / Edge），不做在线部署 |
| 演示方式 | 视频演讲为主，不提供在线链接 |
| 开发环境 | Windows 本地裸跑 `npm run dev`，不使用 WSL、不使用 Docker |

### 为什么是纯前端 + 薄后端而非"只有一个页面"

项目复杂度不在横向铺页面，而在纵向叠层次。绘图工具天然是单画布形态。背后有五层：

```
语音采集层  →  语义编译层  →  场景图/状态层  →  执行渲染层  →  模型调度层
```

一个页面背后有五层，这是深，不是简单。

## 三、技术架构

| 层 | 选型 | 说明 |
|---|---|---|
| 前端框架 | **React + TypeScript + Vite** | `react-konva` 生态成熟，Zod/TS 类型集成顺，社区资料与 AI 辅助准确度最高 |
| 语音识别（ASR） | **后端百度 ASR 主路径 + 浏览器 Web Speech 备用** | T5 起新增后端 ASR：前端录音，后端藏百度 key 并调用短语音识别 REST API；单段音频强制小于 60 秒。Web Speech 仅作开发/备用模式。状态机见 `大脑设计.md` §7 |
| 意图解析 | LLM 调用 | 自然语言 → 结构化 JSON（动作/图形/坐标/颜色/数量/目标） |
| 画布执行层 | **Konva.js + react-konva** | 对象模型清晰、与场景图一一映射；Fabric.js 的鼠标交互能力在纯语音场景中不是重点 |
| 状态管理 | **Zustand**（场景图存前端内存） | 数据与渲染解耦，避免 re-render 卡顿 |
| 前后端通信 | **REST（P0）+ SSE（P1）** | 两天内先用 REST 跑通闭环；思考流用前端阶段状态模拟，SSE 作为有余力再上的增强 |
| 后端 | **Express 薄代理**（本地 Node 服务） | 仅藏 key + 模型路由 + 故障转移，不含业务逻辑；Vite 通过 `/api` 代理到 Express |

### 工程形态（已拍板）

```text
/client  React + TypeScript + Vite + react-konva
/server  Express + 模型 Provider Adapter + ASR Provider Adapter
npm run dev  使用 concurrently 同时启动 Vite 与 Express
```

不采用 Next API 或 Vite middleware 作为 P0：项目不需要服务端渲染，Vite middleware 虽然能少起一个进程，但调试边界更绕。Express + Vite proxy 更直白、更适合两天内稳定交付。

### T5 后端 ASR 补强（新增决策）

浏览器 Web Speech API 在当前本机环境出现漏听和 `network` 错误，因此正式演示链路不再把它作为唯一 ASR 依赖。新增语音输入分层：

```
麦克风录音  →  前端录音窗口/静音检测  →  /api/asr  →  百度 ASR  →  transcript  →  /api/parse
```

- **后端藏 key**：百度 `API Key` / `Secret Key` 只在 `/server/.env`，前端只上传音频。
- **短指令策略**：百度短语音识别单次请求限制在 60 秒以内；MVP 默认 8–12 秒短指令窗口，55 秒硬停止上传。
- **格式策略**：优先统一为 16kHz、单声道、PCM/WAV；如浏览器输出格式不同，转换逻辑集中在录音/ASR adapter。
- **失败降级**：百度未配置、网络失败或音频无效时，不执行旧 transcript；前端提示可重试，并可按配置降级到 Web Speech / 转写测试输入。
- **开发入口收束**：“执行层调试”“转写测试输入”保留到开发期，但正式模式通过 `VITE_ENABLE_DEV_TOOLS=false` 隐藏。

### 性能注意（避免卡顿）

1. 不要每次状态变化就全量重绘整个画布，用库的局部更新/分层。
2. 不要把场景图塞进 React state 后频繁深拷贝触发全树 re-render。
3. 原则：**React/状态管"数据真相"，Konva 管"画布渲染"，两者解耦**。几百个对象无压力。

## 四、模型方案（已确认）

资源：拥有 GPT-5.2 全家桶、Claude 全家桶、DeepSeek V4.0 Flash/Pro 的 API。

**最终方案：Claude 主用 + DeepSeek 自动兜底 + 前端可手动切换 + key 全在后端。**

| 决策点 | 内容 |
|---|---|
| 主模型 | Claude（复杂指令拆解、严格 JSON、遵循长 prompt 最强） |
| 备用模型 | DeepSeek V4.0（国内访问最稳、最便宜，作为高可用兜底） |
| 默认来源 | 后端配置的 key，前端永远拿不到 key |
| 前端切换 | 前端只发 `model` 参数（如 `"claude" | "deepseek"`），后端按参数路由；**切的是"用哪个模型"，不是"填哪个 key"** |
| 自动故障转移 | Claude 调用失败/超时时，后端自动 fallback 到 DeepSeek，前端无感 |

### 模型 SDK / Provider Adapter

后端统一封装一个模型接口，前端永远只调 `/api/parse`：

```ts
interface ModelProvider {
  parseInstruction(input: ParseInput): Promise<ParseResult>;
}
```

- `ClaudeProvider`：优先使用官方 Anthropic SDK / 官方 HTTP API。
- `DeepSeekProvider`：按 OpenAI-compatible HTTP 格式调用。
- `FallbackProvider`：默认先 Claude；失败、超时或 JSON 不合法时自动切 DeepSeek。
- `.env` 支持官方 API key，也预留 `BASE_URL` 适配中转服务；接入方式变化不影响前端和 DSL。

写进设计文档的亮点表述：
> 主模型 Claude 保证拆解质量，DeepSeek 作为高可用兜底，自动故障转移保证服务不中断。

## 五、交互与界面（已确认）

### 题目边界的理解（演示路径 100% 语音）

题目约束："用户不能使用鼠标或键盘，仅通过语音指令完成绘图创作。"明确界定如下：

- **绘图创作**（画图形、改样式、移动、删除、组合、撤销、清空）：**唯一合法路径是语音**，演示全程不碰鼠标键盘。
- **meta 控制**（暂停/继续监听、切模型）：**同时支持语音和侧边栏按钮**，但侧边栏按钮仅作为**开发期调试 / 故障恢复入口**，不出现在正式演示路径里。正式演示一律用语音指令完成 meta 控制。
- **首次麦克风授权**属于"启动准备"（等价于打开 app、点允许权限），不属于"绘图创作"，不违反约束。

> 视频里可明确口播："除首次授权麦克风外，全程未使用鼠标或键盘。"把质疑彻底堵死。

### 暂停 / 继续监听的纯语音闭环（关键）

不能真停掉 ASR，否则就听不到"继续"。采用**三档监听电源态**（详见大脑设计 §7 ASR 状态机）：

- `active`：正常监听，全量解析绘图指令。
- `standby`：暂停态——**麦克风和语音识别仍在运行**，但只对唤醒词（"继续/开始绘图"）做本地匹配响应，其余转写一律忽略、不调 LLM、不画图。说"暂停监听"进入此态，说"继续"回到 active。
- `off`：唯一真正关麦克风的硬关闭态，仅用于隐私场景，演示不使用。

这样"暂停→继续"全程语音可达，不存在"停了就听不到继续"的悖论。

### 界面布局

全屏干净界面：

- **中央**：大画布（唯一的绘图区，纯语音操作）
- **侧边栏**：
  - 实时思考流（核心亮点，见下）
  - 当前监听状态（**三档电源态**：🟢 active 全量解析 / 🟡 standby 仅唤醒词 / ⚫ off 硬关闭）
  - 暂停 / 继续按钮（鼠标按钮仅作开发/故障恢复入口；正式演示走语音"暂停监听"/"继续"）
  - 当前模型显示 + 切换
  - （可选）指令历史

### 监听电源态（解决"纯语音暂停悖论"）

"暂停后就听不到'继续'"是个逻辑悖论。解法是把监听分三档，**standby 不关 ASR**：

| 状态 | 麦克风/ASR | 行为 | 进入方式 |
|---|---|---|---|
| 🟢 active | 开 | 全量解析，正常绘图 | "继续" / 启动 |
| 🟡 standby | **仍开** | 只匹配唤醒词（"继续/开始绘图"），其余忽略，不调 LLM | "暂停监听" / "停一下" |
| ⚫ off | 关 | 真正停麦克风，需鼠标重启（仅隐私场景，演示不用） | 鼠标点击 |

唤醒词匹配在本地对 final transcript 做字符串判断，**零成本零延迟**，纯语音闭环不破。

### 实时思考流（务必做）

把藏在后面的智能层**可视化**给评委看，是把深度"亮"出来的最佳手段：

```
🎤 正在听… 
→ 📝 识别到："画三个由小到大的红圆" 
→ 🧠 正在拆解（调用 Claude）… 
→ 📐 已生成 3 个图元，计算布局… 
→ ✅ 已绘制
```

它让"大脑在工作"肉眼可见，原本评委看不到的语义解析、拆解、路由过程全部暴露在眼前。

## 六、核心深度点（评分重点，精力都押这里）

1. **复杂指令拆解**：
   - "画个房子" → 墙 + 屋顶 + 窗 + 门一组图元，作为一个 **group** 整体管理（可整体移动/删除/缩放）
   - "五个由小到大、彩虹色的圆排成一排" → 循环 + 布局 + 渐变
2. **定位模型（LLM 出意图，代码算像素）**：四选一统一定位 `absolute / anchor / relative / layout`。LLM 只输出"空间意图"，确定性布局器算最终坐标，稳定且美观。
3. **上下文指代与相对操作**（基于场景图）：
   - "把它变大一点"、"刚才那个圆改成蓝色"、"把最大的那个删掉"
4. **容错与澄清对话**：
   - 模糊/口误指令 → AI 主动反问澄清，而非乱画；置信度低时降级处理
   - 危险指令（清空 / 删除全部）→ **语音二次确认**（pendingAction，用户说"确定"才执行）
5. **（加分）流式响应降延迟 + 自测评测表**：
   - 简单/复杂/模糊指令各 N 条，报告识别准确率、拆解成功率、平均延迟。带数据 = 有严谨设计。

## 七、Key 管理（简化版）

不做公开在线链接，防刷体系整体砍掉，只保留铁律：

| 场景 | key 放哪 | 进仓库吗 |
|---|---|---|
| 本地开发 | `.env` 文件 | 否（`.gitignore` 忽略） |
| 仓库代码 | 只读 `process.env`，提供 `.env.example` 占位 | 是，但无真值 |
| 别人本地跑 | 他自己的 key | 否 |

- 铁律：真实 key 绝不进公开仓库，只通过环境变量运行时注入。
- 本地跑时 `.env` 里有 key，正常使用、正常录视频。
- 百度 ASR 同样遵循此规则：`BAIDU_ASR_API_KEY` / `BAIDU_ASR_SECRET_KEY` 只放后端 `.env`，前端不得出现真实凭据。
- （可选，5 分钟）厂商后台设消费上限兜底。
- **已砍掉**：在线部署、来源校验、限流、用户自带 key 输入框。
