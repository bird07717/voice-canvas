## 阶段 0 — 架构图与工程初始化（Day1 上午前置）

> 目标：把空仓库变成一个能跑起来、目录清晰、key 安全的工程骨架。这一阶段不写业务逻辑，只搭地基。

### T0.1 画整体架构图（标清自研层 vs 框架层）⬜

- **目标**：一张图说清"框架负责手和眼，自研层负责大脑"，作为设计文档配图和视频素材。
- **步骤**：
  1. 用 Mermaid 画五层数据流：语音采集 → 语义编译 → 场景图/状态 → 执行渲染 → 模型调度。
  2. 明确标注哪些是框架（Web Speech / Konva / LLM SDK），哪些是自研（场景图、定位模型、DSL、拆解 prompt、布局器、ASR 状态机）。
  3. 存为 `docs/架构图.mermaid` 或 `docs/架构图.md`。
- **验收**：图中能一眼区分自研层与框架层；与 `大脑设计.md` §0 数据流一致。
- **文档锚点**：MVP方案 §二、§十；大脑设计 §0。
- **优先级**：P0（叙事价值高，先做不亏）。

### T0.2 初始化 monorepo 工程骨架 ⬜

- **目标**：建立 `/client` + `/server` 双目录结构，能各自独立启动。
- **步骤**：
  1. 根目录 `npm init`，建 `/client`（Vite + React + TS）和 `/server`（Express + TS）。
  2. `/client`：`npm create vite@latest client -- --template react-ts`，装 `konva react-konva zustand zod`。
  3. `/server`：装 `express cors dotenv`，TS 配置（`tsx` 或 `ts-node-dev` 做热重载）。
  4. 根 `package.json` 装 `concurrently`，写 `npm run dev` 同时起两个进程。
  5. Vite 配 `/api` 代理到 Express 端口（如 `localhost:3001`）。
- **验收**：`npm run dev` 一条命令同时起 Vite（前端）和 Express（后端）；浏览器能打开空白 React 页，前端 `fetch('/api/health')` 能打到后端返回的探活响应。
- **文档锚点**：MVP方案 §三「工程形态」；大脑设计 §11。
- **优先级**：P0。

### T0.3 配置 key 管理与环境变量 ⬜

- **目标**：真实 key 绝不进仓库，运行时注入。
- **步骤**：
  1. `/server` 建 `.env`（放真实 key）和 `.env.example`（占位）。
  2. 根 `.gitignore` 忽略 `.env`、`node_modules`、`dist`。
  3. 后端只读 `process.env.ANTHROPIC_API_KEY` / `process.env.DEEPSEEK_API_KEY`，预留 `*_BASE_URL` 适配中转。
- **验收**：`git status` 不出现 `.env`；`.env.example` 含全部需要的键名但无真值。
- **文档锚点**：MVP方案 §七。
- **优先级**：P0（安全铁律，越早越好）。

### T0.4 首次 git 提交 ⬜

- **目标**：建立基线，之后每个里程碑提交一次。
- **步骤**：确认 `.gitignore` 生效后 `git add` + 首次 commit（如 `chore: 工程骨架与文档基线`）。
- **验收**：`git log` 有第一条提交；提交内不含任何 key 或 `node_modules`。
- **优先级**：P0。

## 阶段 1 — Konva 执行层闭环（Day1 上午）

> 目标：先用按钮手动触发，把"场景图 → Konva 渲染"这条确定性管线跑通。这一层是 dumb 且可靠的"手"，不涉及任何 LLM。先有手，才谈大脑。

### T1.1 定义场景图数据结构与 Zustand store ⬜

- **目标**：实现 `SceneObject` / `Group` / `SceneState`，作为渲染唯一真相源。
- **步骤**：
  1. 按 `大脑设计.md` §1.2/§1.3/§1.4 定义 TS 类型：`SceneObject`（id/type/geometry/style/label/groupId/seq/z）、`Group`（id/label/memberIds/bbox/seq）、`SceneState`（objects/groups/lastCreatedIds/focusIds/canvas/pendingAction/nextSeq）。
  2. 用 Zustand 建 store 持有 `SceneState`，**数据与渲染解耦**（不把场景图塞进 React state 频繁深拷贝）。
  3. 实现 bbox 推导工具（由各 type geometry 算 `{x,y,w,h}`，§1.2 约定）。
- **验收**：能在 store 里手动塞一个 circle 对象，类型检查通过。
- **文档锚点**：大脑设计 §1。
- **优先级**：P0。

### T1.2 Konva 画布渲染层 ⬜

- **目标**：场景图变化时，Konva 画布正确渲染对应图形。
- **步骤**：
  1. `react-konva` 建全屏 `Stage` + `Layer`，画布尺寸按 `canvas.width/height`。
  2. 写 `<SceneRenderer>`：遍历 `objects`，按 type 渲染 Circle/Rect/Triangle/Line/Text/Ellipse。
  3. **统一中心点定位**：rect/triangle 的 Konva 左上角 = `x - w/2, y - h/2`，circle 直接用中心（§1.5）。
  4. 用库的局部更新/分层，避免每次全量重绘（性能注意点）。
- **验收**：store 里塞 3 个不同图形，画布正确显示，位置符合中心定位约定。
- **文档锚点**：大脑设计 §1.5；MVP方案 §三「性能注意」。
- **优先级**：P0。

### T1.3 实现 create / setStyle / transform / delete 执行器 ⬜

- **目标**：实现核心 operation 的确定性 apply 逻辑（先不接 LLM，直接喂 JSON）。
- **步骤**：
  1. 实现 `applyOperation(op, state)`：`create`（新建对象，分配 seq/id）、`setStyle`（改 fill/stroke/...）、`transform`（move/moveTo/scale/resize/rotate）、`delete`。
  2. 每次执行更新 `focusIds` / `lastCreatedIds` / `nextSeq`。
  3. 颜色归一化：中文 → 十六进制色表（§3.6）。
- **验收**：在调试面板手动构造 operation JSON，能正确创建、改色、移动、删除对象。
- **文档锚点**：大脑设计 §3.3、§5、§3.6。
- **优先级**：P0。

### T1.4 撤销 / 重做（历史快照栈）⬜

- **目标**：实现 `undo` / `redo`。
- **步骤**：
  1. 每次成功执行操作前，深拷贝 `objects + groups` 压撤销栈（§1.4）。
  2. `undo` 弹栈恢复并把当前态压重做栈；`redo` 反向。
  3. group 操作以"整组"为一次撤销步。
- **验收**：画 3 个图形后连续撤销/重做，画布状态精确回退/恢复。
- **文档锚点**：大脑设计 §1.4、§3.3。
- **优先级**：撤销 P0 / 重做 P1。

### T1.5 开发调试面板（按钮手动触发）⬜

- **目标**：在没接语音前，用按钮触发上述全部操作，验证执行层。
- **步骤**：侧边栏临时放按钮："画圆""画方块""改色""左移""删除""撤销""清空"，各自构造一个 operation 数组喂执行器。
- **验收**：纯靠点按钮能完成 L1+L2+L3（撤销/清空）的基础闭环。
- **说明**：这些按钮是开发期/故障恢复入口，**不出现在正式演示路径**（MVP方案 §五）。
- **优先级**：P0（开发脚手架）。

### T1.6 阶段提交 ⬜

- **验收**：`git commit`（如 `feat: Konva 执行层闭环，按钮可驱动基础图形`）。能演示"点按钮画图改色移动删除撤销"。

## 阶段 2 — 语音最小闭环（Day1 下午）

> 目标：打通"说话 → 画图"。接 Web Speech API（带 ASR 状态机雏形）+ 后端 LLM 代理 + Zod 校验。做完这一阶段，"画一个红色的圆"能靠嘴完成。

### T2.1 Zod 严格 schema（响应包络 + Operation 判别联合）⬜

- **目标**：把 `大脑设计.md` §3 的 schema 用 Zod 完整落地，运行时校验 + TS 类型二合一。
- **步骤**：
  1. 实现 `Geometry`（discriminatedUnion on `shape`）、`Style`、`Position`（四选一 on `mode`）。
  2. 实现 `Operation`（discriminatedUnion on `op`：create/createGroup/setStyle/transform/delete/clear/undo/redo）。
  3. 实现 `ResponseEnvelope`（understanding/operations/reply/clarify）。
- **验收**：合法 JSON 通过校验，非法字段被拒并能定位错误。
- **文档锚点**：大脑设计 §3.1–§3.3。
- **优先级**：P0。
- **可并行**：与 T2.2 后端代理可并行开发。

### T2.2 后端 LLM 代理（ModelProvider + Claude）⬜

- **目标**：前端只调 `/api/parse`，后端注入 key 调 Claude 返回严格 JSON。
- **步骤**：
  1. 定义 `ModelProvider.parseInstruction(input): Promise<ParseResult>`。
  2. 实现 `ClaudeProvider`（Anthropic 官方 SDK/API），把 system prompt + 紧凑场景图 + transcript 组装请求。
  3. `/api/parse` 接收 `{transcript, scene, groups, recentTurns, canvasSize, model}`，返回 LLM 的 JSON。
  4. **安全提示**：这是本地代理，无鉴权可接受（演示用本地裸跑）；若日后公开需加访问控制。
- **验收**：用 curl/Postman 发一条 transcript，后端返回合法的响应包络 JSON。
- **文档锚点**：MVP方案 §四；大脑设计 §0、§11。
- **优先级**：P0。

### T2.3 拆解 system prompt 落地 ⬜

- **目标**：把 `大脑设计.md` §4.1 prompt 骨架 + §4.2 few-shot 写成实际 prompt 模板。
- **步骤**：
  1. 实现紧凑场景图序列化（§1.6 单行文本格式，token 降 60–70%）。
  2. 把画布信息、定位说明、当前场景图、最近 3 轮对话、核心规则、few-shot 示例 A–F 拼成 system message。
- **验收**：空画布下"画一个红色的圆"返回与示例 A 一致结构的 JSON。
- **文档锚点**：大脑设计 §4、§1.6。
- **优先级**：P0。

### T2.4 接 Web Speech API + ASR 状态机雏形 ⬜

- **目标**：实现语音识别，区分 partial/final，搭起状态机骨架。
- **步骤**：
  1. 封装 Web Speech API（中文识别），首次请求麦克风授权。
  2. 实现状态机核心态：`idle / listening(active) / parsing / executing / speaking`（§7.1）。
  3. **partial 只用于思考流展示，绝不触发解析**；只有 final 进入 finalizing → parsing（§7.2）。
  4. `speaking`（TTS 播报）时抑制识别防回声自触发。
- **验收**：说话时侧边栏实时显示 partial 文本；final 触发一次解析；播报时不会把自己的声音当输入。
- **文档锚点**：大脑设计 §7。
- **优先级**：P0。

### T2.5 串联完整闭环（说话 → 解析 → 校验 → 执行）⬜

- **目标**：把 T2.1–T2.4 接成一条线。
- **步骤**：final transcript → 组装 `/api/parse` 请求 → 收到 JSON → Zod 校验 → 执行器 apply → 更新场景图 → TTS 播 `reply`。
- **验收**：对着麦克风说"画一个红色的圆""把它变成蓝色""往左移""撤销"，画布正确响应。
- **文档锚点**：大脑设计 §0、§5。
- **优先级**：P0（这是 Day1 的核心里程碑）。

### T2.6 阶段提交 ⬜

- **验收**：`git commit`（如 `feat: 语音→画图最小闭环打通`）。能演示纯语音画出第一个图形。

## 阶段 3 — 大脑深度层（Day2 上午）

> 目标：把项目从"语音查表 demo"拉升到"有上下文记忆和复杂拆解能力的语义引擎"。这一阶段是评分重点。

### T3.1 确定性布局器：anchor 九宫格 + relative 四方向 ⬜

- **目标**：实现 LLM 出意图、代码算像素的核心。
- **步骤**：
  1. `anchor`：九宫格等分点 + `dx/dy` 微调 + `margin` 防贴边（§2.2）。
  2. `relative`：解析 `ref`（含同批次 `tempId → 真实 id`），按 side（left/right/above/below）放外侧 gap 处，另一轴居中（§2.3）。支持负 gap 用于房子门窗嵌入。
  3. 兜底链：relative 引用失败 → anchor center → absolute 中心 → 边界 clamp（§2.5）。
- **验收**："在右上角画个圆""在圆的左边画个方块"位置正确。
- **文档锚点**：大脑设计 §2.2、§2.3、§2.5。
- **优先级**：P0。

### T3.2 确定性布局器：layout（row + grid）⬜

- **目标**：批量布局成员，做后处理 pass。
- **步骤**：
  1. `row`：总宽 = Σ成员宽 + gap×(count−1)，整排水平居中，align 决定 y（§2.4）。
  2. `grid`：未指定列数时 `cols = ceil(sqrt(count))`，单元格取成员最大 bbox + gap。
  3. 同 `layoutId` 的 ops 收集后统一计算各自中心坐标。
- **验收**："画五个由小到大的圆排成一排""画一个 3x3 网格圆"布局整齐居中。
- **文档锚点**：大脑设计 §2.4；指令清单 L5。
- **优先级**：P0。

### T3.3 Group 对象与组合体整体操作 ⬜

- **目标**：让"房子/车/太阳"成为一等公民，可整体移动/缩放/删除。
- **步骤**：
  1. 实现 `createGroup` + 带 `groupId` 的 create，建立 group 索引（成员 id + 缓存 bbox）。
  2. group 级操作传播规则（§1.3 表）：move 整组平移、scale 以 bbox 中心等比、delete 删全部成员+记录、setStyle 按语义。
  3. 成员变动即重算 bbox，撤销以整组为单位。
- **验收**："画一个房子"生成墙+屋顶+门+窗一组（示例 D），"把那个房子往左移"整组移动。
- **文档锚点**：大脑设计 §1.3、§4.2 示例 D。
- **优先级**：P0。

### T3.4 复杂指令拆解联调 ⬜

- **目标**：验证 prompt + 布局器 + group 协同，覆盖 L5 核心。
- **步骤**：用演示脚本逐条测："在右边画房子"、"五个由小到大彩虹圆排成一排"、"3x3 网格圆"、"在房子左边画个圆"。调 prompt few-shot 直到稳定。
- **验收**：L5 的 P0 条目（语义组合体/数量循环/渐变排列/网格/相对定位）全部可用。
- **文档锚点**：指令清单 L5；大脑设计 §4.2。
- **优先级**：P0。

### T3.5 上下文指代与相对操作（L4）⬜

- **目标**：实现"它/刚才的/最大的/第二个/红色的那个"指代消解。
- **步骤**：
  1. 主路径：紧凑场景图 + 最近 3 轮对话喂 LLM，由它出 `targetIds`（§1.7 第一层）。
  2. 兜底：代码确定性规则（上一个→lastCreatedIds、它→focusIds、最大/最小→bbox 排序、那个房子→group.label，§1.7 第二层）。
  3. 相对位移/缩放走模糊量词约定表（§3.5）。
- **验收**："把它变大""把最大的那个删掉""刚才那个圆改成绿色""往左移一点"全部命中正确对象。
- **文档锚点**：大脑设计 §1.7、§3.5；指令清单 L4。
- **优先级**：P1（深度点，时间允许必做）。

### T3.6 模型路由与故障转移（FallbackProvider）⬜

- **目标**：Claude 主用，失败自动兜底 DeepSeek，前端可手动切。
- **步骤**：
  1. 实现 `DeepSeekProvider`（OpenAI-compatible HTTP）。
  2. 实现 `FallbackProvider`：默认 Claude，超时/失败/JSON 不合法时自动切 DeepSeek。
  3. 前端只发 `model` 参数，后端按参数路由（切模型不切 key）。
- **验收**：手动让 Claude key 失效，请求自动 fallback 到 DeepSeek，前端无感；"切换到 DeepSeek"语音/按钮可切。
- **文档锚点**：MVP方案 §四；大脑设计 §8.3。
- **优先级**：故障转移 P0 / 手动切换 P2。
- **可并行**：与 T3.5 可并行（不同人/不同文件）。

### T3.7 阶段提交 ⬜

- **验收**：`git commit`（如 `feat: 布局器/group/上下文指代/模型兜底`）。能演示房子拆解、渐变圆排列、上下文指代、模型自动兜底。

## 阶段 4 — 容错、思考流与收尾（Day2 下午）

> 目标：补齐题目硬指标"容错性"，把智能层可视化（思考流），打磨 UI，完成交付物。

### T4.1 危险指令二次确认（pendingAction，P0-lite）⬜

- **目标**：清空/删除全部不会因一句口误误删整画。
- **步骤**：
  1. 执行层遇 `clear` / delete 全部对象 → 不执行，设 `pendingAction`，语音播报"确定要清空吗？说『确定』继续"。
  2. 下一句**前端本地词匹配**（不调 LLM）：命中确认词→执行；其他→取消并正常处理这句（§6.2）。
- **验收**："清空画布" → AI 反问 → 说"确定"才清空，说别的则取消。
- **文档锚点**：大脑设计 §6.2；指令清单 L6。
- **优先级**：P0-lite。

### T4.2 澄清对话协议（clarify）⬜

- **目标**：模糊/歧义指令主动反问而非乱画。
- **步骤**：
  1. LLM 解析不出唯一目标 → `operations: []` + `clarify: {question, options}`（§3.1、示例 F）。
  2. 前端展示问题 + 语音播报，把澄清问答并入 recentTurns，下一句重新解析（§6.1）。
- **验收**：多对象时说"把那个东西弄红"→ AI 反问"你是指圆还是方块？"→ 回答后正确执行。
- **文档锚点**：大脑设计 §6.1、§4.2 示例 F；指令清单 L6。
- **优先级**：P1。

### T4.3 容错兜底（缺参/口误/无法理解/越界/非法）⬜

- **目标**：覆盖 L6 其余条目。
- **步骤**：缺参数用默认值（P0）；口误模糊匹配（P1）；无关话礼貌提示（P1）；坐标越界 clamp（P2）；删除不存在对象友好提示（P1）；部分成功策略按 ExecReport 播报（§3.4）。
- **验收**："画个圆"无颜色用默认色；"删掉第十个"（只有3个）友好提示不存在。
- **文档锚点**：大脑设计 §3.4、§5；指令清单 L6。
- **优先级**：缺参推断 P0 / 其余 P1–P2。

### T4.4 standby 暂停监听闭环 ⬜

- **目标**：纯语音暂停/继续，不破闭环。
- **步骤**：
  1. 实现 `standby` 态：mic/ASR 仍开，只对唤醒词（继续/开始绘图）本地匹配（§7.2）。
  2. "暂停监听/停一下"→ standby；"继续/开始绘图"→ active。
  3. 撤销/确认词等同走本地快速通道（§8.2）。
- **验收**：说"暂停监听"后只响应"继续"，其余转写忽略不画图；说"继续"恢复。
- **文档锚点**：大脑设计 §7、§8；指令清单 L3。
- **优先级**：P0。

### T4.5 实时思考流侧边栏 ⬜

- **目标**：把大脑工作过程可视化（核心亮点）。
- **步骤**：映射管线各阶段：🎤正在听 → 📝识别到「…」 → 🧠调用{模型}拆解中 → 📐生成N个操作/算布局 → ✅已执行/❓需澄清/⚠️等待确认（§0、MVP方案 §五）。由前端 ASR 状态机阶段状态驱动（REST P0，思考流用阶段状态模拟，掩盖等待感）。
- **验收**：每说一句，侧边栏按阶段推进显示；调用模型、生成操作数、布局计算肉眼可见。
- **文档锚点**：大脑设计 §0、§8.5；MVP方案 §五。
- **优先级**：P1（最佳"亮深度"手段，强烈建议做）。

### T4.6 UI 打磨与监听状态显示 ⬜

- **目标**：全屏干净界面，三档电源态可视。
- **步骤**：中央大画布；侧边栏含思考流、监听状态（🟢active/🟡standby/⚫off）、当前模型显示+切换、暂停/继续按钮（仅开发/故障恢复）、可选指令历史。
- **验收**：界面整洁、状态清晰，适合录视频。
- **文档锚点**：MVP方案 §五「界面布局」「监听电源态」。
- **优先级**：P1。

### T4.7 自测评测表（加分项）⬜

- **目标**：用数据证明严谨设计。
- **步骤**：准备简单/复杂/模糊指令各 N 条，跑一遍记录识别准确率、拆解成功率、平均延迟，整理成表。
- **验收**：`docs/` 出一份评测数据表。
- **文档锚点**：MVP方案 §六.5、§八.3。
- **优先级**：P2（有余力做，叙事价值高）。

### T4.8 交付物：文档 + 视频 + 仓库 ⬜

- **目标**：完成三件交付物。
- **步骤**：
  1. **设计文档**：回填 `指令能力清单.md` 实现统计表（计划/已实现/完成率），写未完成项及原因 + 成本/延迟优化技巧 + 评测数据。
  2. **README**：本地运行步骤（`npm run dev`）、`.env.example` 说明、架构简介。
  3. **演示视频**：按演示脚本（指令清单 §演示脚本）由浅入深拍，强调"全程未碰鼠标键盘（除首次授权麦克风）"。
  4. 最终 commit + 确认仓库 key 不泄露。
- **验收**：仓库可被他人 clone 后按 README 跑起来；视频覆盖 L1→L6 亮点。
- **文档锚点**：MVP方案 §八；指令清单 §演示脚本、§实现情况统计。
- **优先级**：P0（交付物本身）。
