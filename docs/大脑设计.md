# 项目大脑设计 —— 场景图 / 指令 DSL / 拆解 Prompt

> 本文档定义 Voice-Canvas 的核心智能层（"大脑"）。它是把"用户口语"编译为"画布操作"的语义引擎，是整个项目区别于查表式 demo 的关键所在。
> 框架（ASR / Konva / LLM）负责"听、画、推理"的原始能力，本文档定义的四件事——**场景图、定位模型、指令 DSL、拆解 Prompt**——是任何框架都不提供、必须自研的部分。
>
> **v2 修订说明**：本版根据架构 review 落实六项补强——纯语音 standby 监听、定位模型（absolute/anchor/relative/layout 四选一 + 确定性布局器）、真正的 group 对象、Zod 判别联合严格 schema（拆掉混杂的 update）、ASR 状态机、危险指令二次确认（P0-lite）。修订点在文中以 **[v2]** 标注。
>
> **v3 / T5 修订说明**：新增后端百度 ASR provider 规划。Web Speech 不再是正式演示唯一识别链路；ASR 层统一输出 transcript，后续语义编译、场景图、执行层契约不变。

## 1. 场景图数据结构（Scene Graph）

### 1.1 设计目标

场景图不只是"图形列表"，它要同时承担四个职责：

1. **渲染真相**：画布上现在有什么，是渲染的唯一数据源。
2. **指代上下文**：序列化后喂给 LLM，让它能解析"它 / 那个 / 最大的"。
3. **关系与时序**：支持"最后画的""最大的""左边那个""那个房子"等查询。
4. **撤销/重做**：通过状态快照栈实现历史回退。

### 1.2 单个对象（SceneObject）

```jsonc
{
  "id": "obj_5",            // 唯一 id，递增生成
  "type": "circle",         // circle | rect | triangle | line | text | ellipse
  "geometry": {             // 几何信息，按 type 不同字段不同（见 §1.7 各 shape schema）
    "x": 600, "y": 400,     // 【中心点】坐标（统一约定，见 §1.5）
    "radius": 50            // circle 专有
  },
  "style": {
    "fill": "#ff0000",      // 填充色，统一存十六进制
    "stroke": "#000000",    // 描边色，可为 null
    "strokeWidth": 0,
    "opacity": 1
  },
  "label": "红色的圆",       // 语义标签（用户怎么称呼它），辅助指代
  "groupId": "grp_1",       // 所属 group（如"房子"的部件），可为 null
  "seq": 5,                 // 创建序号，用于"最后/上一个/刚才"
  "z": 5                    // 图层顺序
}
```

> **包围盒（bbox）** 不直接存，由各 type 的 geometry 在序列化时统一推导出 `{x, y, w, h}`（左上角+尺寸），用于"最大/最小/最左"这类查询。避免数据冗余与不一致。

### 1.3 Group 对象（复合体）[v2 新增]

"房子""车""太阳"这类语义组合体，必须有**一等公民的 group 记录**，否则"那个房子往左移"只能靠零散对象瞎推断。

**设计原则：对象（SceneObject）是唯一真相源；group 是建在对象之上的"索引 + 语义封装"。** group 不重复存几何，只引用成员 id，bbox 由成员推导并缓存。

```jsonc
{
  "id": "grp_1",
  "label": "房子",            // 用户怎么称呼它（指代用）
  "memberIds": ["obj_3","obj_4","obj_5","obj_6","obj_7"],
  "bbox": { "x": 720, "y": 270, "w": 300, "h": 320 },  // 缓存值，成员变动时重算
  "seq": 1,                  // 创建序号，用于"刚才那个房子"
  "createdAt": 1700000000
}
```

**group 级操作的传播规则（执行层确定性处理）：**

| 操作作用在 groupId 上 | 传播到成员的行为 |
|---|---|
| `transform.move / moveTo` | 整组同向平移，重算 bbox |
| `transform.scale` | 以 group bbox 中心为基准，所有成员几何与相对位置等比缩放，重算 bbox |
| `setStyle` | 默认作用于"主体成员"或全部成员（按指令语义，见 prompt 规则）|
| `delete` | 删除所有成员 + group 记录，整组算一次撤销步 |

> bbox 缓存采用"成员变动即重算"策略，不存脏数据。撤销以"整组"为单位压栈。

### 1.4 场景状态（SceneState）

```jsonc
{
  "objects": [ /* SceneObject[] */ ],
  "groups":  [ /* Group[] */ ],            // [v2] 复合体索引
  "lastCreatedIds": ["obj_5", "obj_6"],    // 上一条指令新建的对象（"刚才那些"）
  "focusIds": ["obj_5"],                   // 当前焦点对象（连续操作时的"它"）
  "canvas": { "width": 1200, "height": 800 },
  "pendingAction": null,                   // [v2] 危险指令待确认（见 §6）
  "nextSeq": 7
}
```

`history`（撤销栈）独立维护：每次成功执行操作前，把当前 `objects + groups` 深拷贝压栈；"撤销"即弹栈恢复。

### 1.5 坐标系约定（统一规则）

| 约定 | 说明 |
|---|---|
| 原点 | 画布左上角 (0,0) |
| 方向 | x 向右增大，y 向下增大（Konva 原生） |
| 定位锚点 | **所有图形以"中心点"定位**：`(x,y)` 是包围盒中心 |
| 画布中央 | 对任意图形都是 `(width/2, height/2)`，如 (600,400) |
| 执行层转换 | rect/triangle 的 Konva 左上角 = `x - w/2, y - h/2`；circle 直接用中心 |

> 统一中心定位是个关键决策：让定位/布局推理"居中、靠左、右上角"时心智模型一致，显著降低坐标计算错误率。

### 1.6 喂给 LLM 的紧凑序列化（重要的成本优化）

直接把全量 JSON 塞给 LLM 既费 token 又慢。改用**单行紧凑文本格式**，信息密度高、LLM 解析无压力：

```
对象：
#3 circle "红色的圆" 中心(400,300) r=50 fill:红 [属于 grp_1]
#4 rect   "蓝方块"   中心(150,150) 80x80   fill:蓝
#5 text   "标题"     中心(600,80)  "我的画" 字号40 fill:黑
组合体：
grp_1 "房子" 成员[#3,...] 包围盒 中心(850,430) 300x320
```

- `#N` 即对象 id（`obj_N` 简写），LLM 输出 targetIds 时回填 `obj_N`；group 用 `grp_N`。
- 颜色用中文（LLM 友好），执行层负责中↔十六进制转换。
- 画布为空时，传一行：`（画布为空）`。

**优化效果**：相比全量 JSON，token 量约降 60–70%，直接降低延迟与成本。写进设计文档"成本优化"章节。

### 1.7 指代消解机制（指代 → id）

这是上下文能力的核心。分两层：

**第一层（主）：LLM 解析。** 把紧凑场景图（含 groups）+ 最近对话喂给 LLM，由它把"它/那个/刚才的/最大的红圆/那个房子"解析为具体 `targetIds`（对象 id 或 group id）。LLM 有完整语言上下文，这是最自然的方案。

**第二层（兜底）：代码确定性规则。** 当 LLM 给不出明确 id 或场景图为空时：
- "上一个/刚才" → `lastCreatedIds`
- "它/这个"（无其他线索） → `focusIds`，若为空则取 `seq` 最大者
- "全部/所有" → 所有 objects
- "最大/最小" → 按 bbox 面积排序取极值
- "那个房子/车" → 按 group.label 匹配
- 解析不出唯一目标 → 触发 **clarify**（不猜）

## 2. 定位模型（Positioning Model）[v2 核心重构]

### 2.1 设计理念：LLM 出"空间意图"，代码算"像素坐标"

旧设计让 LLM 直接算所有坐标，复杂布局（整齐网格、并排、相邻）不稳定，clamp 只能防越界、不能保证美观。**v2 改为：LLM 只表达空间意图，确定性布局器负责把意图编译成精确像素。** 智能在 LLM，确定性与美观在代码。

`position` 是 `create` 和 `transform.moveTo` 上的字段，**四选一判别联合**（按 `mode` 区分）：

```jsonc
// (1) absolute —— 直接坐标（简单场景兜底，LLM 仍可用）
{ "mode": "absolute", "x": 600, "y": 400 }

// (2) anchor —— 九宫格锚点 + 可选偏移（"右上角""画布中央"）
{ "mode": "anchor", "region": "top-right", "dx": 0, "dy": 0 }

// (3) relative —— 相对某对象定位（"在房子旁边""圆的右边"）
{ "mode": "relative", "ref": "grp_1", "side": "right", "gap": 40 }

// (4) layout —— 批量布局成员（"五个圆排成一排"）
{ "mode": "layout", "layoutId": "L1", "type": "row", "index": 0, "count": 5, "gap": 30, "align": "middle" }
```

### 2.2 anchor 九宫格（P0）

`region` 取值（九宫格）：

```
top-left      top-center      top-right
middle-left   center          middle-right
bottom-left   bottom-center   bottom-right
```

布局器算法：九宫格在画布上取等分点作为该区域中心，图形中心对齐该点；`dx/dy` 为可选微调（像素）。画布留 `margin`（默认 60px）防止贴边。

### 2.3 relative 相对定位（P0，四方向）

`side ∈ { left, right, above, below }`，`ref` 可以是真实对象/group id，也可以是**同一批 operations 内 create 提供的 `tempId`**；`gap` 为间距（默认 40px），`dx/dy` 为最终微调偏移。
布局器先解析本批次 `tempId → 真实 id` 映射，再取 `ref` 的 bbox，按 side 把新图形放到其外侧 gap 处，另一轴居中对齐，最后叠加 `dx/dy`。P0 允许负 `gap` 用于房子门窗这类嵌入式部件；更语义化的 `inside/on-top-of` 放 P1。
> P1 扩展：`inside / on-top-of`（叠放）、八方向、`around`（环绕）。

### 2.4 layout 布局器（P0：row + grid）

同一批 create 共享 `layoutId`，布局器对这组 ops 做**后处理 pass**：根据 `type / count / gap / align / origin` 和每个成员的实际尺寸，计算各自中心坐标。

- **row（横向并排）**：总宽 = Σ成员宽 + gap×(count−1)；整排默认在画布水平居中（或以 origin 为起点）；各成员中心 x 累加推进，y 由 align（top/middle/bottom）决定。
- **grid（网格）**：未指定列数时 `cols = ceil(sqrt(count))`；按行列均匀铺开，单元格尺寸取成员最大包围盒 + gap。
- `origin` 可选，指定整体布局的锚点（默认画布中心区域）。

> P1 扩展：`column`、`circle`（环形）、复杂 align、不等距分布。这些未完成项写进设计文档"未完成原因"。

### 2.5 兜底链

定位解析失败（如 relative 引用了不存在的对象）→ 回退 anchor center → 仍失败 → absolute 画布中心。任何模式算出的最终坐标都过一遍**边界 clamp**（防超出画布），clamp 退化为最后一道防线而非主力。

## 3. 指令 DSL / JSON Schema（Zod 判别联合）[v2 重构]

### 3.1 响应包络（Response Envelope，唯一契约）

LLM **每次调用都必须**输出且仅输出这个 JSON（无任何额外文字）：

```jsonc
{
  "understanding": "把上一个圆放大",   // 一句话复述理解（思考流展示 + debug）
  "operations": [ /* Operation[] */ ], // 要执行的操作列表，可为空
  "reply": "好的，已经把圆放大了",      // 给用户的语音回应（执行成功时）
  "clarify": null                      // 需要澄清时填对象，否则 null
}
```

**Zod 定义（执行层运行时校验 + TS 类型，二合一）：**

```ts
const ResponseEnvelope = z.object({
  understanding: z.string(),
  operations: z.array(Operation),          // 见 §3.3
  reply: z.string().nullable(),
  clarify: z.object({
    question: z.string(),
    options: z.array(z.string()).optional(),
  }).nullable(),
});
```

歧义时：`operations: []`，`reply: null`，`clarify: { question, options? }`。

### 3.2 几何与样式 schema（按 shape 区分）[v2 明确化]

```ts
// —— 各 shape 的 geometry（不含位置，位置由 §2 的 position 决定）——
const CircleGeo   = z.object({ shape: z.literal("circle"),   radius: z.number().positive() });
const RectGeo     = z.object({ shape: z.literal("rect"),     width: z.number().positive(), height: z.number().positive() });
const TriangleGeo = z.object({ shape: z.literal("triangle"), width: z.number().positive(), height: z.number().positive() });
const EllipseGeo  = z.object({ shape: z.literal("ellipse"),  radiusX: z.number().positive(), radiusY: z.number().positive() });
const LineGeo     = z.object({ shape: z.literal("line"),     points: z.array(z.number()).min(4) }); // 相对中心的点
const TextGeo     = z.object({ shape: z.literal("text"),     content: z.string(), fontSize: z.number().positive() });
const Geometry = z.discriminatedUnion("shape", [CircleGeo, RectGeo, TriangleGeo, EllipseGeo, LineGeo, TextGeo]);

// —— 样式（与几何彻底分离，杜绝旧版 update.set 混用）——
const Style = z.object({
  fill: z.string().nullable().optional(),         // 中文或十六进制
  stroke: z.string().nullable().optional(),
  strokeWidth: z.number().min(0).optional(),
  opacity: z.number().min(0).max(1).optional(),
});

// —— 位置（§2 四选一）——
const Region = z.enum([
  "top-left","top-center","top-right",
  "middle-left","center","middle-right",
  "bottom-left","bottom-center","bottom-right",
]);

const LayoutOrigin = z.object({
  region: Region,
  dx: z.number().optional(),
  dy: z.number().optional(),
});

const Position = z.discriminatedUnion("mode", [
  z.object({ mode: z.literal("absolute"), x: z.number(), y: z.number() }),
  z.object({ mode: z.literal("anchor"), region: Region, dx: z.number().optional(), dy: z.number().optional() }),
  z.object({ mode: z.literal("relative"), ref: z.string(), side: z.enum(["left","right","above","below"]),
             gap: z.number().optional(), dx: z.number().optional(), dy: z.number().optional() }),
  z.object({ mode: z.literal("layout"), layoutId: z.string(), type: z.enum(["row","grid"]), index: z.number().int(),
             count: z.number().int(), gap: z.number().optional(), align: z.enum(["top","middle","bottom"]).optional(),
             origin: LayoutOrigin.optional(), cols: z.number().int().positive().optional(), rows: z.number().int().positive().optional() }),
]);
```

### 3.3 操作类型（Operation，判别联合 on `op`）[v2 拆掉混杂 update]

旧版 `update.set` 把几何字段和样式字段混在一起，职责不清。v2 拆成**职责单一**的操作：

```ts
const Operation = z.discriminatedUnion("op", [

  // 创建图形
  z.object({
    op: z.literal("create"),
    geometry: Geometry,
    style: Style.optional(),
    position: Position,
    label: z.string(),
    tempId: z.string().optional(),       // 本批 operations 内临时引用名，如 "wall"，用于 relative.ref
    groupId: z.string().optional(),     // 属于某组合体时填
  }),

  // 创建组合体（房子/车/太阳）：先建组，成员用带 groupId 的 create
  z.object({
    op: z.literal("createGroup"),
    groupId: z.string(),                // 如 "grp_new1"，执行层映射为真实 id
    label: z.string(),
  }),

  // 只改样式（fill/stroke/strokeWidth/opacity）
  z.object({
    op: z.literal("setStyle"),
    targetIds: z.array(z.string()).min(1),
    style: Style,
  }),

  // 只改几何（移动/缩放/缩放到/旋转）—— 四选一
  z.object({
    op: z.literal("transform"),
    targetIds: z.array(z.string()).min(1),
    action: z.discriminatedUnion("kind", [
      z.object({ kind: z.literal("move"),   dx: z.number(), dy: z.number() }),          // 相对平移
      z.object({ kind: z.literal("moveTo"), position: Position }),                       // 移到某位置
      z.object({ kind: z.literal("scale"),  factor: z.number().positive() }),            // 等比缩放
      z.object({ kind: z.literal("resize"), width: z.number().positive(), height: z.number().positive() }),
      z.object({ kind: z.literal("rotate"), deg: z.number() }),
    ]),
  }),

  // 删除
  z.object({ op: z.literal("delete"), targetIds: z.array(z.string()).min(1) }),

  // 危险：清空（执行层拦截二次确认，见 §6）
  z.object({ op: z.literal("clear") }),

  // 历史
  z.object({ op: z.literal("undo") }),
  z.object({ op: z.literal("redo") }),
]);
```

> 注意：**暂停/继续监听不是 LLM operation**——它走前端本地快速通道（§7、§8），不经过模型。

### 3.4 单操作执行结果 & 部分成功策略 [v2 新增]

执行层逐条 apply，每条产出结果，整体允许"部分成功"：

```ts
type OpResult =
  | { status: "ok"; affectedIds: string[] }
  | { status: "skipped"; reason: string }     // 如 targetId 不存在
  | { status: "failed";  reason: string };    // 如几何非法

// 整批执行返回
interface ExecReport { results: OpResult[]; okCount: number; failCount: number; }
```

**部分成功策略**：执行所有合法操作，跳过非法/无目标的；`reply` 由前端按报告动态调整——全成功播 LLM 的 `reply`；部分失败播"完成了 N 个，有 M 个没找到对象"。绝不因一条失败而整批回滚（除非是 group 原子操作）。

### 3.5 模糊量词 → 数值的约定

| 口语 | 默认量化 |
|---|---|
| 大一点 / 小一点 | scale.factor 1.3 / 0.77 |
| 大很多 / 小很多 | scale.factor 1.8 / 0.55 |
| 往左/右/上/下 移 | move ±100px |
| 往左移一点 | move ±40px |
| 往左移很多 | move ±250px |

> 写进 prompt 后 LLM 稳定遵循；也允许用户明确指定（"放大 3 倍""往右移 200"）覆盖默认。

### 3.6 颜色处理

- LLM 输出可用中文（红/蓝/浅绿）或十六进制。
- 执行层维护**中文→十六进制色表**（红=#ff0000、天蓝=#87ceeb…），统一转换。
- 无法识别的颜色 → 触发 clarify 或回退默认色并提示。

## 4. 复杂指令拆解 Prompt（System Prompt 设计）

设计原则：**角色单一、格式强约束、给足上下文、用示例教拆解与定位、明确边界规则。**

### 4.1 Prompt 骨架（可直接用作 system message）

```
你是「语音绘图助手」的语义编译器。你的唯一职责：把用户的中文口语指令，
结合当前画布状态，翻译成严格的 JSON 绘图操作。除 JSON 外不要输出任何文字。

# 画布信息
- 尺寸：宽 {W} × 高 {H} 像素
- 坐标系：左上角为原点 (0,0)，x 向右增大，y 向下增大
- 所有图形以「中心点」定位

# 定位（重要：你只表达"空间意图"，由程序计算最终像素，不要自己硬算复杂坐标）
每个 create / moveTo 都要带 position，从四种 mode 选一：
- absolute：你直接给 {x,y}（仅用于简单的单个图形）
- anchor：九宫格区域 {region: top-left|top-center|top-right|middle-left|center|
          middle-right|bottom-left|bottom-center|bottom-right, dx?, dy?}
- relative：相对某对象 {ref: 对象/组id, side: left|right|above|below, gap?}
- layout：批量布局 {layoutId, type: row|grid, index, count, gap?, align?}
  → 多个图形排列时，给同一 layoutId、各自 index、相同 count，由程序算坐标

# 当前画布对象（场景图）
{compact_scene}

# 最近对话（最多 3 轮）
{recent_turns}

# 可用操作类型
create / createGroup / setStyle / transform / delete / clear / undo / redo
（字段定义见下方示例；setStyle 只改颜色描边，transform 只改几何，二者不要混用）

# 输出格式（必须是单个合法 JSON，无 markdown、无解释文字）
{
  "understanding": "一句话复述你对指令的理解",
  "operations": [ ...操作数组，可为空... ],
  "reply": "执行成功时给用户的简短中文回应；需澄清时为 null",
  "clarify": null 或 { "question": "...", "options": ["...","..."] }
}

# 核心规则
1. 【指代消解】"它/这个/那个/刚才的/最大的红圆/那个房子"→ 结合场景图与对话解析出
   具体 id（对象 obj_N 或组 grp_N），填入 targetIds。无法唯一确定 → operations 留空 + clarify。
2. 【定位用意图】简单单图形可用 absolute 或 anchor；"在X旁边"用 relative；
   "N 个排成一排/网格"用 layout（同 layoutId + 各 index）。不要为复杂布局手算一堆坐标。
3. 【组合体拆解】"房子/车/太阳"等：先 createGroup 建组，再用带同一 groupId 的多个 create
   画出各部件。需要在同一批 operations 里互相引用时，给关键部件加 tempId（如 wall），
   其他部件用 relative.ref 指向这个 tempId；不要用 label 当 ref。
4. 【模糊量词】"大一点"=scale 1.3，"往左移"=move 左 100px…（按约定表，可被明确数值覆盖）。
5. 【职责分离】改颜色用 setStyle；移动/缩放/旋转用 transform；不要把样式塞进 transform。
6. 【歧义不猜】指代不明、信息缺失到无法执行 → operations 留空 + clarify 提问。
7. 【不自作主张】只画/改用户明确要求的，不擅自加装饰。
8. 【危险操作】"清空/全部删掉"→ 输出 clear（程序会再做一次语音确认，你不必自己确认）。
```

## 5. 执行层与校验（确定性的"手"）

LLM 输出不可全信，执行前必须校验。执行层纯确定性：

1. **Zod 校验**：响应包络 + 每个 operation 过 schema。不合法的操作丢弃并记录 `failed`。
2. **定位/布局解析**：把 position 意图编译为像素坐标（§2 布局器）；layout 组做后处理 pass。
3. **targetId 存在性**：引用了不存在的 id → 该操作 `skipped`，必要时转 clarify。
4. **边界 clamp**：最终坐标超出画布 → 拉回；尺寸非法 → 修正。
5. **颜色归一化**：中文/非法颜色 → 查色表转十六进制，失败回退默认。
6. **危险拦截**：`clear`（及"删除全部"）→ 不直接执行，进入 pendingAction 二次确认（§6）。
7. **执行 + 更新场景图**：逐条 apply 到 Konva，同步更新 objects/groups/focusIds/lastCreatedIds，压撤销栈。
8. **结果反馈**：按 ExecReport 决定语音播报（全成功 / 部分成功，§3.4）。

> "LLM 负责理解、代码负责把关"——这层校验是项目鲁棒性的体现，也是设计文档里可写的工程点。

## 6. 容错、澄清与危险指令确认

### 6.1 澄清协议

| 情况 | 处理 |
|---|---|
| 指代不明（多个候选） | clarify 提问，列候选，等用户语音回答 |
| 指令含糊（缺关键信息） | clarify 追问，或用合理默认值并告知 |
| ASR 明显口误/乱码 | LLM 识别为无意义 → reply 友好提示"没太听清，能再说一遍吗" |
| 引用对象不存在 | 该操作 skipped + 提示"画布上没找到那个对象" |
| 用户回答澄清问题 | 把澄清问答并入 recentTurns，重新走一次解析 |

澄清问答进入 `recentTurns`，形成多轮上下文。

### 6.2 危险指令二次确认（P0-lite）[v2 新增]

**仅覆盖 `clear`（清空画布）和"删除全部"**，机制极简、确定性，不依赖 LLM：

```
执行层遇到 clear / delete(全部对象)
  → 不执行，设置 pendingAction = { type:"clear", ops:[...] }
  → 语音播报 + 思考流："确定要清空画布吗？说『确定』继续，说『取消』放弃"
下一句用户输入（前端本地判断，不调 LLM）：
  → 命中确认词（确定/是的/对/继续）→ 执行 pendingAction，清空 pendingAction
  → 其他任何输入 → 取消 pendingAction，按正常流程处理这句话
```

> 成本极低（一个状态 + 一次本地词匹配），但显著提升"容错"叙事：危险操作不会因一句口误而误删整张画。其他删除（删单个/某几个对象）不走二次确认，保持流畅。

## 7. ASR 状态机（语音输入侧的"地基"）[v2 新增，T5 扩展]

ASR 层只负责把语音变成 transcript。可靠的纯语音体验需要一个明确的状态机来管理：浏览器 Web Speech 模式直接产出 final transcript；后端百度 ASR 模式先录音上传，再由 `/api/asr` 返回 transcript。两种模式都汇入同一条 `finalizing → parsing → executing` 链路。

### 7.1 状态与迁移

```
        ┌────────────────────────── "暂停监听" ──────────────────────────┐
        ▼                                                                 │
 ┌──────────┐  mic 授权   ┌────────────────┐ transcript ready ┌──────────┐│
 │   idle   │ ──────────▶ │ listening      │ ───────────────▶ │finalizing││
 │(未启动)  │             │ /recording     │                  │(去抖/校验)│
 └──────────┘             └────────────────┘                  └────┬─────┘
        ▲                        ▲   ▲                              │
        │ "硬关闭"               │   │ "继续"                       │ 有效绘图指令
        │                  ┌─────┴───┴──────┐                  ┌────▼─────┐
   (hard off)              │   standby      │◀── "暂停监听" ───│ parsing  │
                           │ (仅唤醒词,mic 开)│                 │(调 LLM)  │
                           └────────────────┘                  └────┬─────┘
                                                                    │
                                          ┌────────┐  ┌──────────┐  │
                                          │speaking│◀─│executing │◀─┘
                                          │(TTS,   │  │(应用操作)│
                                          │ 抑制听)│  └──────────┘
                                          └───┬────┘
                                              │ 播报完
                                              ▼ 回到 listening
```

| 状态 | 含义 | mic / ASR |
|---|---|---|
| `idle` | 未启动（授权前 / 硬关闭后） | 关 |
| `listening(active)` | 正常监听，全量解析绘图指令 | 开 |
| `recording` | 后端 ASR 模式下正在采集单句音频 | 开 |
| `uploading` | 后端 ASR 模式下上传音频到 `/api/asr` | 可开，暂停处理新句 |
| `transcribing` | 等待百度 ASR 返回 transcript | 可开，暂停处理新句 |
| `finalizing` | 拿到 final，去抖、过滤、校验是否值得解析 | 开 |
| `parsing` | 调用 LLM 解析中 | 开（但不处理新 final，排队/丢弃） |
| `executing` | 应用操作到画布 | 开 |
| `speaking` | TTS 播报回应 | **抑制识别**（防回声自触发） |
| `standby` | 暂停态：**mic 仍开、ASR 仍跑**，但只对唤醒词响应 | 开 |
| `asrError` | ASR 配置、网络、格式或空音频失败 | 可重试或降级 |

### 7.2 关键策略

- **partial vs final**：浏览器 ASR 的 `partial`（中间结果）只用于**思考流实时展示**"正在听…"，**绝不触发解析**；后端 ASR 没有稳定 partial 时，只展示录音/上传/识别阶段；只有最终 transcript 才进入 finalizing。
- **standby 不关 ASR**：这是纯语音闭环的关键——"暂停监听"进入 standby，麦克风和识别仍在跑，但前端只对唤醒词（"继续/开始绘图/醒醒"）做本地匹配，其余转写一律忽略、不调 LLM、不画图。说"继续"→ 回 active。彻底避免"停了就听不到继续"的悖论。
- **硬关闭（off→idle）**：唯一真正关麦克风的状态，仅用于隐私场景；由于 ASR 已关闭，需鼠标或系统权限流程重新开启。演示不使用。
- **静默超时**：active 下持续 N 秒无 final，思考流回到"待命"提示，不做动作。
- **去抖 / 回声抑制**：speaking 状态抑制识别；finalizing 对 Nms 内重复/高度相似的 final 去重，避免同一句被识别两次重复执行。
- **低置信度 / 空结果**：浏览器 ASR 置信度低于阈值，或后端 ASR 返回空 transcript → 不直接解析，走澄清"没太听清，能再说一遍吗"。
- **噪声/过短过滤**：final 词数过少或纯语气词 → 忽略，防噪声误触发。
- **危险指令**：解析结果含 clear → 进入 §6.2 的 pendingAction（与状态机正交）。

### 7.3 后端 Baidu ASR 模式（T5）

后端 ASR 只替换“听”的实现，不改变“想”和“画”的协议。核心链路：

```
MediaRecorder/WebAudio 录音
  → 前端限制单段时长（默认 8–12s，硬上限 55s）
  → POST /api/asr
  → 后端获取/缓存百度 access_token
  → 调用百度短语音识别 REST API
  → 返回 transcript
  → 进入现有 /api/parse + 执行层
```

**设计约束：**

1. **60 秒上限内建到产品逻辑**：百度短语音识别单段音频限制在 60 秒以内，前端以 55 秒作为硬停止，避免边界误差。
2. **短指令优先**：Voice-Canvas 的交互是“一句话一个绘图意图”，不是会议转写；MVP 不做长音频流式听写。
3. **音频格式集中处理**：优先转换/提交 16kHz、单声道、PCM/WAV。格式转换封装在 ASR adapter，不散落到业务组件。
4. **后端藏凭据**：`BAIDU_ASR_API_KEY` / `BAIDU_ASR_SECRET_KEY` 只存在后端 `.env`，前端不得出现真实 key 或 token。
5. **失败不执行**：百度 ASR 报错、空结果、超时或格式不合法时，不复用上一次 transcript，不进入 `/api/parse`，只提示用户重试或切换 ASR provider。
6. **开发工具可隐藏**：转写测试输入和执行层调试保留给开发排障，正式模式通过 `VITE_ENABLE_DEV_TOOLS=false` 隐藏。

## 8. 延迟与成本优化（写进设计文档的成本章节）

1. **场景图紧凑序列化**：单行文本替代全量 JSON，token 降 60–70%。
2. **本地快速通道（不调 LLM）**：暂停/继续监听、撤销、清空确认词、唤醒词 → 前端本地正则/词匹配处理，**零延迟零成本**；只有绘图/编辑指令才调模型。
3. **主备模型 + failover**：默认 Claude（拆解质量），失败自动切 DeepSeek（高可用）；可按指令复杂度路由简单指令到 DeepSeek Flash（更便宜）。
4. **REST P0，SSE P1**：两天内先用普通 REST 返回完整 JSON，思考流由前端阶段状态模拟（正在听→正在解析→校验→布局→执行）；SSE/流式 token 作为 P1 增强，不阻塞核心闭环。
5. **思考流即时反馈**：识别到文本立刻在侧边栏显示（partial），解析与执行阶段由前端状态推进，掩盖 LLM 调用等待感，提升体感流畅度。
6. **最近对话只保留 3 轮**：控制上下文长度，平衡理解力与成本。
7. **定位交给布局器而非 LLM 反复试**：减少因坐标算错导致的重试/澄清往返。
8. **ASR provider 可替换**：浏览器 ASR 与后端百度 ASR 都统一输出 transcript，后续 `/api/parse` 与执行层不关心来源，降低排障和切换成本。

## 9. 关键设计决策汇总

| # | 决策 | 取舍理由 |
|---|------|---------|
| 1 | 场景图即上下文，LLM 无状态 | 状态留前端，LLM 每次拿全量场景，架构简单、可控 |
| 2 | 指代消解归 LLM，执行归代码 | 智能与确定性分离，执行层 dumb 且可靠 |
| 3 | **定位 = LLM 出意图 + 代码布局器算像素** [v2] | 复杂布局稳定、美观；clamp 仅作兜底 |
| 4 | **group 为对象之上的索引，对象是唯一真相源** [v2] | 支持组合体整体操作，又不冗余存几何 |
| 5 | **Zod 判别联合 + 拆分 setStyle/transform** [v2] | 运行时校验 + 类型安全；职责单一杜绝字段混用 |
| 6 | 统一中心点定位 | 空间推理心智一致，坐标错误率低 |
| 7 | 歧义不猜，主动澄清 | 直接命中题目"容错性" |
| 8 | **危险指令 pendingAction 二次确认（P0-lite）** [v2] | 防误删整画，成本极低，容错叙事强 |
| 9 | **ASR 状态机 + standby 监听** [v2] | 纯语音闭环逻辑自洽，不存在"停了听不到继续" |
| 10 | 紧凑序列化 + 本地快速通道 | 同时优化延迟与成本，成本控制叙事的实证 |
| 11 | **REST P0，SSE P1** | REST 先保证闭环简单稳定；SSE 只作为思考流增强，不让工程复杂度吞掉核心能力 |
| 12 | **后端 Baidu ASR provider** [T5] | 绕开 Web Speech 的网络/兼容性不稳定，把 ASR 凭据和厂商调用收束到后端 |
