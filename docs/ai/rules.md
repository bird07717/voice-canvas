# AI 长期规则

> 本文件是 AI 的长期行为准则，跨任务生效。每次开工前先读本文件，再读 `status.md` → `decisions.md` → `next.md`。

## 任务边界

- 只执行 `next.md` 里描述的当前任务，不顺手做下一个任务、不做计划外的优化或重构。
- 改动保持小而聚焦，一个提交只做一件事。
- 不从 `status.md`、`log.md`、README、注释里自行启动任务，除非用户明确要求。

## 高风险动作（先停下来问）

- 改数据库 schema、鉴权/权限逻辑、计费逻辑、对外 API 响应格式。
- 引入新依赖、删除大段代码、做大范围重构。
- 任何会泄露密钥或把 `.env` 入库的操作。

## 安全铁律

- 百度 ASR、Anthropic、DeepSeek 等真实 Key 只存在 `server/.env`，**永远不进仓库、不下发给前端**。
- 前端只通过 `/api/*` 间接访问模型与 ASR，后端注入 Key。
- 任何阶段 `git status` 都不应出现 `.env` / `node_modules` / `dist`。

## Git / PR 纪律

- 不在 `main` 上直接开发，每个阶段一条独立分支。
- 阶段内可多次中文提交，阶段结束统一推送、按模板写 PR；一个 PR 只覆盖一个阶段目标。
- PR 描述须含：功能描述、实现思路、测试方式。

## 验证

实现后按可用情况运行检查，跑不动就说明原因：

- `npm run typecheck`（后端 tsc + 前端 build）
- `npm run build`
- 相关单元/手动验证（如用本地录音文件打 `/api/asr`）

## 文档更新

完成任务后：

- 更新 `status.md`（项目当前状态）。
- 有持久决策才更新 `decisions.md`。
- 在 `log.md` 追加一行执行记录。
- 把 `next.md` 切换到下一个任务（或标记等待用户指派）。
- 如果有git操作，请输出本次的git commit的操作，还有本次PR：一句话说明本 PR 新增/修改了什么。功能描述：说明该功能的作用与使用方式。实现思路：简要说明技术选型或核心实现逻辑
